# Documentation Standards

## Цель

Этот документ задаёт единый стандарт корневой документации workspace и документов, которые агент создаёт в рамках общей исследовательской программы.

## Основные принципы

- Документы должны быть структурированными и пригодными для продолжения работы в новой сессии.
- Записи должны быть достаточными для научной трассируемости.
- Нужно отделять факты, интерпретации, гипотезы и решения.
- Если информация критична, она должна жить в markdown-файле, а не только в чате.

## Типы корневых документов

### Операционные

- `AGENTS.md`
- `DOCUMENTATION_INDEX.md`
- `PROJECT_MAP.md`
- `BASELINE_INTAKE_SPEC.md`
- `KAGGLE_AUTONOMOUS_WORKFLOW.md`
- `SOURCE_COLLECTION_POLICY.md`

### Память и история

- `USER_PROMPTS_LOG.md`
- `RESEARCH_JOURNAL.md`
- `AGENT_ACTIVITY_LOG_SPEC.md`

### Стандарты

- `DOCUMENTATION_STANDARDS.md`
- `EXPERIMENT_LOGGING_STANDARD.md`
- `MATH_RESEARCH_WORKFLOW.md`

## Рекомендуемая структура нового md-документа

```text
# Title

## Purpose
...

## Scope
...

## Main content
...

## Risks / limitations
...

## Next actions
...
```

## Статусы

Рекомендуемые статусы:

- `planned`
- `in_progress`
- `confirmed`
- `partial`
- `needs_verification`
- `archived`
- `rejected`

Для русскоязычных narrative-документов можно использовать:

- `запланировано`
- `в работе`
- `подтверждено`
- `частично`
- `требует проверки`
- `архив`
- `отклонено`

## Правило источников

Если утверждение опирается на внешний источник, документ должен явно показывать:

- что это за источник;
- как он был использован;
- является ли он baseline reference, доказательным источником или merely inspiration.

## Правило фактов и интерпретаций

Не смешивать:

- факт из кода;
- факт из документации;
- факт из лога эксперимента;
- вывод агента;
- гипотезу для будущей проверки.

Если нужно, использовать явные маркеры:

- `Fact`
- `Inference`
- `Hypothesis`
- `Decision`

## Правило избыточности

Так как workspace ориентирован на будущую научную работу, допустима умеренная избыточность, если она:

- сохраняет контекст;
- облегчает восстановление цепочки решений;
- не дублирует бессмысленно уже существующие блоки текста.

## Обязательные обновления

После крупных изменений агент должен проверить:

- нужно ли обновить `DOCUMENTATION_INDEX.md`;
- нужно ли добавить запись в `RESEARCH_JOURNAL.md`;
- нужно ли зафиксировать новый промпт в `USER_PROMPTS_LOG.md`;
- нужно ли обновить локальные project docs.

## Именование

- Корневые стандарты и policy documents именуются в uppercase snake-like style: `*_SPEC.md`, `*_POLICY.md`, `*_WORKFLOW.md`.
- Narrative journals и maps именуются описательно: `RESEARCH_JOURNAL.md`, `PROJECT_MAP.md`.
- Не использовать двусмысленные названия вроде `notes.md` для корневых важных документов.
