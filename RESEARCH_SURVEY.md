# Research Survey

## Цель обзора

Этот документ фиксирует первый глубокий аудит двух проектов внутри workspace `D:\Agentic_Solve_Math`. Он нужен как стартовая долговременная память перед разработкой системы автоматического исследования Kaggle-задач и математических направлений при помощи LLM-агентов.

Обзор отвечает на вопросы:

- что уже есть в каждом проекте;
- какие практики стоит поднять на корневой уровень;
- какие пробелы мешают автономной агентной работе;
- как разделить роли между двумя проектами без потери структуры и воспроизводимости.

## Метод исследования

При подготовке обзора были изучены:

- ключевые README и навигационные файлы обоих проектов;
- workflow-документы, правила для агентов, журналы изменений и индексы;
- структура каталогов, entry points и runners;
- документы по гипотезам, мультимодельному анализу, экспериментам и логам;
- representative code files, показывающие зрелость reusable parts.

## Итог верхнего уровня

Workspace уже содержит почти все ключевые идеи, нужные для будущего автономного research agent, но они распределены между двумя проектами:

- `Math_Hypothese_AutoCheck_Witch_Agents/` уже содержит зрелую модель **исследовательской памяти и анализа математических гипотез при помощи агента**.
- `ML in Math/` уже содержит зрелую модель **инженерной дисциплины и экспериментального цикла вокруг pancake-проблемы**.

Главная задача корневого слоя не в том, чтобы переписать эти проекты, а в том, чтобы:

1. сделать их роли явными;
2. добавить общую память и трассируемость;
3. стандартизовать intake новых Kaggle-задач;
4. обязать агента сохранять промпты и действия в форме, пригодной для научной работы.

## Проект A: `Math_Hypothese_AutoCheck_Witch_Agents/`

### Текущее назначение

Проект позиционируется как лаборатория систематического анализа математических гипотез при помощи LLM-агента. Архитектурно он строится вокруг идеи атомарных карточек знаний и вычислительной проверки.

### Главные компоненты

- `docs/` — ядро долгосрочной памяти;
- `src/` — небольшой Python framework для генерации графов, инвариантов и тестирования гипотез;
- `lean/` — формализация гипотез в Lean 4;
- `.cursor/` — правила и агентные настройки;
- `notebooks/` — исследовательские ноутбуки;
- root-level scripts — точечные исследовательские скрипты.

### Что в проекте особенно ценно

#### 1. Сильная модель долговременной памяти

Проект уже умеет мыслить не как папка со скриптами, а как база знаний. Особенно важны:

- карточки гипотез;
- карточки концепций;
- карточки доказательств;
- карточки инсайтов;
- диалоги с LLM;
- индекс и доменные индексы;
- отчёты по отдельным гипотезам.

Это готовый шаблон для корневой научной памяти агента.

#### 2. Есть workflow работы с сырьём

Документация проекта уже формулирует очень полезную идею:

- пользователь может бросать сырые материалы в inbox;
- агент должен распознать их тип;
- затем преобразовать их в нормализованные карточки и отчёты.

Эта логика практически идеально переносится на корневой intake через `baseline/`.

#### 3. Есть мультимодельная схема

Проект уже учитывает, что разные LLM дают разные результаты, и предлагает сохранять их раздельно, а потом синтезировать. Это особенно важно для:

- генерации гипотез;
- анализа статей;
- proof attempts;
- проверки спорных идей.

#### 4. Есть мост к формальной математике

Даже если Lean-часть пока незавершённая, сам факт наличия formalization path очень ценен. Корневой слой должен сохранять ссылку на этот маршрут, чтобы будущие вычислительные наблюдения могли эволюционировать в формальные утверждения.

### Основные слабые места

#### 1. Много документации, мало orchestration code

Почти все правильные процессы уже описаны, но пока не автоматизированы end-to-end. Проект отлично задаёт research discipline, но не даёт универсальный engine автономной обработки новых задач.

#### 2. Малая reusable code base относительно амбиций

Python framework существует, но он довольно компактен. Значительная часть реальной исследовательской работы всё ещё живёт в отдельных скриптах и конкретном кейсе `H-001`.

#### 3. Зрелость неравномерна

Память и структура зрелые, formal proof и автоматизация пока нет. Это значит, что корневой слой должен опираться на документацию проекта как на источник методологии, а не ожидать от него готового универсального рантайма.

### Вывод по проекту A

`Math_Hypothese_AutoCheck_Witch_Agents/` должен считаться каноническим источником для:

- структуры исследовательской памяти;
- карточек знаний;
- мультимодельного анализа;
- математической постановки;
- доказательных и formalization workflows.

## Проект B: `ML in Math/`

### Текущее назначение

Проект представляет собой инженерно-исследовательскую среду для Kaggle/algorithmic optimization. В текущем состоянии он предметно сфокусирован на pancake problem, но архитектурно уже содержит важные паттерны для будущих задач.

### Главные компоненты

- `src/` — модульный код солверов, submission tooling и ML;
- `scripts/runners/` — orchestration и batch experiments;
- `docs/` — навигация, история, гипотезы, анализ результатов;
- `runs/` — артефакты запусков;
- `article/` — заготовка научной статьи;
- `hypotheses/` — отдельная папка для анализа идей агентом;
- `baseline/` — проектные исходники и референсы.

### Что в проекте особенно ценно

#### 1. Есть реальный experiment loop

В отличие от более memory-oriented проекта A, здесь уже есть operational cycle:

- выбрать метод;
- прогнать solve;
- проверить корректность;
- оценить against baseline;
- записать лог;
- обновить результаты и гипотезы.

Это образец того, как корневой агент должен обращаться с engineering experiments.

#### 2. Есть дисциплина обновления документации

Навигационный документ для агентов задаёт очень полезное поведение:

- что читать в начале;
- как не трогать исходные baseline files;
- как логировать эксперименты;
- какие markdown-файлы обновлять после важных запусков.

Это почти готовый operational doctrine для agentic development.

#### 3. Есть журнал изменений и анализ гипотез

Проект уже поддерживает:

- историю изменений;
- список гипотез со статусами;
- документ анализа логов и вывода verdicts;
- явную привязку между экспериментами и narrative-документацией.

Именно такую трассируемость нужно поднять на корневой уровень.

#### 4. Есть реальные runners и артефакты

Наличие `main.py`, `run_best.py`, `run_experiment.py`, `run_research.py`, experiment logs и `runs/` говорит о том, что проект не только описывает эксперименты, но и реально их организует.

### Основные слабые места

#### 1. Узкая предметная специализация

Сейчас проект engineering-wise силён, но domain-wise связан с pancake problem. Поэтому на корне нельзя просто принять его структуру как универсальную для любых Kaggle-задач без адаптации.

#### 2. Внутренний `baseline/` уже занят

Внутренний `ML in Math/baseline/` — это исторический архив и рабочий референс конкретного проекта. Поэтому новый корневой `baseline/` должен быть осмыслен иначе: как intake папка, а не как проектный reference store.

#### 3. Часть логов и артефактов сосуществует с рабочей структурой

Это нормально для активного research repo, но корень должен отделять:

- persistent memory;
- runtime artifacts;
- raw task intake;
- long-term scientific trace.

### Вывод по проекту B

`ML in Math/` должен считаться каноническим источником для:

- инженерного experiment loop;
- структуры runners;
- логирования solve/evaluate;
- сопоставления гипотез и результатов;
- дисциплины обновления документации после прогонов.

### Из проекта A

- Inbox-style обработка сырья.
- Карточки знаний и индексы.
- Разделение гипотез, инсайтов, концепций, доказательств и диалогов.
- Мультимодельное разнесение анализов.
- Явная фиксация промптов как части исследовательского контекста.

### Из проекта B

- Entry point discipline.
- Solve/evaluate/log/update cycle.
- Журнал изменений.

## Главные пробелы на уровне всего workspace

### 1. Нет общего корневого контракта агента

До текущего шага у workspace не было единого `AGENTS.md`, который бы:

- объяснял роль обоих проектов;
- требовал сохранять промпты;
- описывал канонический intake;
- задавал единые правила синхронизации памяти.

### 2. Нет корневого журнала пользовательских промптов

Это критичный пробел, потому что для будущей научной работы важны не только результаты, но и:

- формулировки задач пользователя;
- последовательность уточнений;
- инструкции по стилю и полноте работы;
- связь промптов с конкретными документами и действиями агента.

### 3. Нет общей схемы для новых Kaggle-задач

Имеется инженерный проект для одной задачи и исследовательский проект для математических гипотез, но нет корневого описания pipeline:

- получил markdown со ссылками;
- собрал источники;
- разобрал соревнование;
- создал структуру исследования;
- разделил engineering и math tracks;
- зафиксировал выводы и дальнейшие шаги.

### 4. Нет общего стандарта сверхподробной трассируемости

Оба проекта по-своему уже хороши в документации, но нужно единое правило: любой серьёзный шаг агента должен быть восстановим постфактум.

## Что должен делать корневой слой

Корневой слой должен быть не третьим отдельным research project, а **meta-layer**:

- описывать и связывать два проекта;
- фиксировать ваши промпты и задачи;
- задавать правила сверхподробного логирования;
- принимать новые Kaggle-задачи через `baseline/`;
- задавать routing logic между engineering и mathematical work;
- хранить центральный индекс и исследовательский журнал workspace.

## Рекомендованное разделение обязанностей будущего агента

### Корень

- хранить долговременную память workspace;
- фиксировать пользовательские задачи;
- поддерживать центральные индексы;
- определять, куда направлять следующую работу;
- сохранять cross-project conclusions.

### `ML in Math/`

- собирать и тестировать инженерные решения;
- запускать runners;
- формировать submission artifacts;
- логировать quantitative results;
- обновлять технические результаты и статусы engineering-гипотез.

### `Math_Hypothese_AutoCheck_Witch_Agents/`

- превращать идеи и статьи в структурированные knowledge artifacts;
- формулировать математические гипотезы;
- синтезировать мультимодельные рассуждения;
- связывать вычисления с доказательными маршрутами;
- поддерживать путь к Lean formalization.

## Первичные рекомендации

1. Сделать корень каноническим местом для памяти, правил и user prompt history.
2. Ввести корневой `baseline/` как intake-папку для новых Kaggle-задач.
3. Не смешивать этот intake с `ML in Math/baseline/`.
4. Жёстко обязать агента вести сверхподробный prompt log и activity log.
5. Принять проект A как источник структуры памяти, а проект B как источник engineering discipline.
6. Сразу документировать не только успешные действия, но и ветки, которые не были выбраны, чтобы сохранить научную воспроизводимость agent process.

## Заключение

Workspace уже содержит очень сильную основу для агентного research system, но она была фрагментирована:

- одна часть знает, **как помнить и структурировать исследование**;
- другая знает, **как проводить и документировать эксперименты**.

После введения корневого слоя эти две части смогут работать как единая программа:

- новые задачи будут заходить через корень;
- память и пользовательские промпты будут сохраняться в одном месте;
- инженерные и математические ветки будут расходиться осмысленно;
- результаты будут возвращаться в общую долговременную память, пригодную и для разработки, и для будущей научной публикации.
