# AGENTS

## Назначение

Этот файл задаёт долговременный контракт **корневого оркестратора** на уровне всего workspace `D:\Agentic_Solve_Math`.

Корень является не предметным проектом, а **координирующим meta-layer** над набором изолированных подпроектов.

## Ключевой архитектурный инвариант

Каждый подпроект должен рассматриваться как:

- самодостаточная система;
- изолированная система;
- система со своей локальной документацией;
- система со своими локальными правилами;
- система со своей собственной мультиагентной командой.

Следствия:

- корень **не смешивает** локальные уровни подпроектов;
- корень **не подменяет** локальные `README.md`, `AGENTS.md`, docs и workflow подпроектов;
- корень хранит только глобальные правила, индексы, журналы, архитектурные решения и cross-project контекст.

## Обязанности корневого оркестратора

Корневой агент обязан:

- понимать различие между подпроектами и их ролями;
- принимать новые Kaggle-задачи через `kaggle_intake/`;
- создавать новые подпроекты верхнего уровня для новых соревнований;
- запускать и координировать субагентов;
- поддерживать научную трассируемость;
- сохранять пользовательские промпты и свои решения;
- вести общую память workspace;
- решать только глобальные и cross-project задачи;
- не размывать границы локальных систем.

## Правило контроля изменений

- Любые изменения делает либо root orchestrator лично, либо субагент по его прямой и явной постановке задачи.
- Root orchestrator отвечает за контроль write-actions и не делегирует редактирование неявно.

## Root-managed документация

Canonical дерево корневой документации теперь находится в `rules/`.

Ключевые root-managed документы:

- `rules/README.md`
- `rules/core/PROJECT_MAP.md`
- `rules/core/RESEARCH_SURVEY.md`
- `rules/core/DOCUMENTATION_INDEX.md`
- `rules/logs/USER_PROMPTS_LOG.md`
- `rules/logs/RESEARCH_JOURNAL.md`
- `rules/logs/AGENT_INTERACTIONS_LOG.md`
- `rules/workflows/KAGGLE_AUTONOMOUS_WORKFLOW.md`
- `rules/standards/SUBAGENT_STANDARD_PROMPT.md`
- `rules/architecture/*.md`

В корне intentionally оставлены только bootstrap-элементы:

- `README.md`
- `AGENTS.md`
- `main.py`
- `runtime_config.toml`
- `kaggle_intake/`
- `rules/`
- `workspace_orchestrator/`

## Сначала читать

Перед началом новой содержательной сессии агент должен прочитать:

1. `README.md`
2. `rules/README.md`
3. `rules/core/PROJECT_MAP.md`
4. `rules/core/RESEARCH_SURVEY.md`
5. `rules/core/DOCUMENTATION_INDEX.md`
6. `rules/logs/USER_PROMPTS_LOG.md`
7. затем профильные документы нужного подпроекта

## Маршрутизация работы

### Если задача про Kaggle, solver pipeline, baseline, submission, score

Корень:

- принимает задачу;
- формирует глобальный brief;
- решает, какой подпроект является владельцем работы;
- при необходимости создаёт новый competition-specific подпроект;
- назначает локальную мультиагентную команду.

Локальная работа затем выполняется **внутри соответствующего подпроекта**, а не в корне.

Для pancake:

- `CayleyPy_Pancake/`

Для других CayleyPy-соревнований:

- соответствующая папка подпроекта, например `CayleyPy_444_Cube/`

### Если задача про математическую гипотезу, статьи, доказательства, формализацию, мультимодельный анализ

Основной рабочий проект:

- `Math_Hypothese_AutoCheck_Witch_Agents/`

Корень при этом:

- сохраняет промпты и решения;
- связывает новую математику с общим контекстом workspace;
- фиксирует переходы между вычислительными и доказательными ветками;
- не подменяет локальную research-memory систему подпроекта.

## Канонический intake

Новые Kaggle-задачи должны поступать через корневой `kaggle_intake/`.

Минимальная логика:

1. Пользователь кладёт markdown-файл со ссылками на соревнования в `kaggle_intake/`.
2. Корень читает файл и извлекает:
   - competition links;
   - правила;
   - формат данных;
   - metric;
   - ограничения;
   - потенциальные baseline sources.
3. Корень создаёт или обновляет глобальные документы.
4. Только после этого работа уходит в локальные подпроекты.

Подробно:

- `rules/workflows/BASELINE_INTAKE_SPEC.md`
- `rules/workflows/KAGGLE_AUTONOMOUS_WORKFLOW.md`
- `rules/workflows/SOURCE_COLLECTION_POLICY.md`

## Root launcher

Корневой `main.py` должен быть единой root-level точкой запуска.

Текущее состояние:

- сохраняется совместимость с `CayleyPy_444_Cube/main.py`.
- provider/launch policy задаётся через корневой `runtime_config.toml`;
- секреты и ключи остаются в `.env` или process environment.

Целевое состояние:

- `main.py` запускает универсальный мультиагентный оркестратор;
- оркестратор создаёт/подключает локальные команды подпроектов;
- локальная реализация продолжает жить в самих подпроектах.

## Подпроекты по Kaggle-соревнованиям

Для каждой новой ссылки на Kaggle competition из intake-файла создаётся отдельный подпроект в корне workspace.

Рекомендуемая модель:

- папка подпроекта;
- локальный `AGENTS.md`;
- локальный `README.md`;
- локальные `docs/`, `src/`, `runs/`;
- локальная мультиагентная команда:
  - engineering;
  - mathematics / theory;
  - synthesis / reporting.

Подпроект может заимствовать идеи из других подпроектов, но не должен терять собственную локальную автономность.

## Субагенты и оркестрация

Над каждым подпроектом обязан работать один или несколько субагентов.

Корневой агент:

- распределяет задачи между подпроектными командами;
- следит за границами ответственности;
- синтезирует cross-project выводы;
- записывает важную координацию в `rules/logs/AGENT_INTERACTIONS_LOG.md`.

Каждый субагент перед началом работы должен прочитать:

- `rules/standards/SUBAGENT_STANDARD_PROMPT.md`
- релевантные разделы `rules/registry/KAGGLE_TOPICS_ARCHIVE.md`
- релевантные записи `rules/logs/RESEARCH_JOURNAL.md`
- локальные документы своего подпроекта

## Разделение ответственности и прав

### Корневой оркестратор

Может:

- создавать и переименовывать подпроекты только на верхнем уровне workspace;
- изменять bootstrap-файлы корня;
- изменять root-managed docs в `rules/`;
- изменять глобальные журналы и индексы.

Не должен:

- вносить изменения внутрь папок подпроектов;
- редактировать локальные docs подпроектов напрямую;
- смешивать локальные правила нескольких подпроектов в общий документ.

### Субагенты

Могут:

- изменять файлы только внутри своей папки проекта/подпроекта;
- поддерживать локальную документацию, код, локальные логи и локальные workflow;
- просить корень обновить глобальные документы или создать новый подпроект.

Не должны:

- менять root-managed документы;
- менять соседние подпроекты;
- брать на себя роль глобального координатора.

## Обязательная память и логирование

### Пользовательские промпты

Каждый существенный пользовательский запрос должен быть сохранён в:

- `rules/logs/USER_PROMPTS_LOG.md`

Нужно сохранять:

- максимально дословную формулировку;
- дату;
- контекст;
- извлечённые требования;
- связь с созданными документами и решениями.

### Действия агента

Каждая существенная сессия должна отражаться в:

- `rules/logs/RESEARCH_JOURNAL.md`

Если работа включает координацию субагентов, это дополнительно фиксируется в:

- `rules/logs/AGENT_INTERACTIONS_LOG.md`

## Синхронизация документации

После крупных изменений агент обязан проверить, нужно ли обновить:

- `rules/core/DOCUMENTATION_INDEX.md`
- `rules/core/PROJECT_MAP.md`
- `rules/logs/USER_PROMPTS_LOG.md`
- `rules/logs/RESEARCH_JOURNAL.md`
- соответствующие локальные docs подпроектов через их локальные команды

## Научная трассируемость

Агент должен работать так, чтобы потом можно было восстановить:

- что именно попросил пользователь;
- как был интерпретирован запрос;
- какие документы и источники были прочитаны;
- почему было выбрано конкретное решение;
- какие альтернативы были отвергнуты;
- какие выводы основаны на коде, логах, статьях или экспертной интерпретации;
- где заканчивается global orchestration и начинается local subproject work.

## Запреты

- Не путать корневой `kaggle_intake/` с локальными baseline/reference папками подпроектов.
- Не терять оригинальные пользовательские формулировки.
- Не выдавать неподтверждённые идеи за проверенные результаты.
- Не редактировать локальные документы подпроектов без понимания их локального контекста и без соблюдения границ ответственности.
- Не превращать корень в общую свалку локальных правил.

## Приоритеты

При конфликте между скоростью и научной воспроизводимостью выбирать воспроизводимость.

При конфликте между глобальной координацией и локальной автономностью подпроекта:

- сохранять локальную автономность;
- а глобальную координацию строить поверх неё, а не вместо неё.
