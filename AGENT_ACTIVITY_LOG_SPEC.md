# Agent Activity Log Spec

## Цель

Этот документ задаёт стандарт сверхподробного логирования действий агента для всего workspace. Он нужен, чтобы любой существенный шаг можно было восстановить для:

- продолжения работы в новой сессии;
- проверки корректности решений;
- подготовки научной статьи;
- последующего анализа стратегии работы агента.

## Принцип

Если действие важно для понимания хода исследования, оно должно быть отражено в markdown-памяти, а не оставаться только в transient chat history.

## Минимальные объекты фиксации

### 1. Пользовательский запрос

Фиксируется в `USER_PROMPTS_LOG.md`.

Обязательные поля:

- дата;
- язык;
- оригинальная формулировка;
- извлечённые требования;
- связь с документами и артефактами.

### 2. Сессия исследования

Фиксируется в `RESEARCH_JOURNAL.md`.

Обязательные поля:

- дата;
- цель сессии;
- что прочитано;
- что создано/изменено;
- какие решения приняты;
- какие риски или открытые вопросы остались.

### 3. Эксперимент или вычислительная проверка

Фиксируется:

- в локальных логах/артефактах соответствующего проекта;
- в `RESEARCH_JOURNAL.md` как narrative summary;
- в профильных docs проекта, если вывод значим.

### 4. Новый источник

Если агент использует статью, Kaggle notebook, discussion thread, репозиторий или иной внешний материал, нужно записать:

- что это за источник;
- почему он релевантен;
- где он используется;
- является ли он подтверждённым фактом, baseline reference или лишь идеей для проверки.

## Обязательная степень подробности

### Кратко писать нельзя, если:

- пользователь дал длинную постановку;
- были архитектурные решения;
- агент выбирал между несколькими маршрутами;
- действия должны потом попасть в научную работу;
- исследование затрагивает оба проекта.

### Достаточно краткости, если:

- изменение строго техническое и локальное;
- шаг не влияет на общую исследовательскую стратегию;
- подробности уже зафиксированы в другом документе и можно сослаться на него.

## Шаблон записи в `RESEARCH_JOURNAL.md`

```text
## YYYY-MM-DD | Session title

### Goal
- ...

### Read / surveyed
- ...

### Decisions
- ...

### Files created or updated
- ...

### Risks / unknowns
- ...

### Next step
- ...
```

## Шаблон фиксации решения агента

Для значимого решения агент должен уметь ответить на пять вопросов:

1. Что решалось?
2. На основе каких документов, логов или источников?
3. Почему выбран именно этот вариант?
4. Какие альтернативы не выбраны?
5. Где зафиксирован результат?

## Требования к пометкам статуса

Во всех summary-документах желательно различать:

- `confirmed` / `подтверждено`
- `partial` / `частично`
- `needs verification` / `требует проверки`
- `rejected` / `отклонено`
- `design decision` / `архитектурное решение`

## Что не считается достаточным логированием

- Короткая финальная фраза в чате без md-следа.
- Необоснованное решение без ссылки на документы.
- Упоминание источника без пояснения, зачем он был нужен.
- Создание файла без записи, почему именно он появился в структуре.

## Приоритетные документы памяти

При любой крупной работе агент должен проверить, нужно ли обновить:

- `USER_PROMPTS_LOG.md`
- `RESEARCH_JOURNAL.md`
- `DOCUMENTATION_INDEX.md`
- профильные markdown-файлы соответствующего проекта

## Главное правило

Если есть сомнение, логировать ли шаг, агент должен логировать его.
