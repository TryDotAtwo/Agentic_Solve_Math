# Experiment Logging Standard

## Цель

Этот документ переносит на корневой уровень лучшие практики experiment logging из инженерного проекта и связывает их с общей долговременной памятью workspace.

## Базовый принцип

Любой значимый эксперимент должен оставлять два следа:

1. **машиночитаемый артефакт** — log, jsonl, csv, metrics file;
2. **человекоориентированную summary-запись** — journal note, result doc или hypothesis update.

## Минимальные поля для эксперимента

Если эксперимент логируется вручную или через код, желательно фиксировать:

- timestamp;
- script or entry point;
- method name;
- input source;
- output artifact;
- score or metric;
- comparison baseline;
- status of correctness check;
- short conclusion.

## Где хранить что

### В профильном проекте

Сырые экспериментальные артефакты должны жить внутри соответствующего проекта:

- engineering run logs — в `ML in Math/` и его `runs/`;
- математические вычислительные артефакты — в `Math_Hypothese_AutoCheck_Witch_Agents/` и его research structure.

### В корневой памяти

В корневых документах нужно хранить:

- summary того, что запускалось;
- зачем запускалось;
- где лежат артефакты;
- какой вывод допустимо считать подтверждённым.

## Правило verification before conclusion

Нельзя объявлять гипотезу подтверждённой только потому, что:

- есть один удачный прогон;
- найден красивый паттерн;
- модель или агент предложили правдоподобное объяснение.

Нужно явно указать:

- что именно проверено;
- на каком наборе данных;
- какая метрика использовалась;
- есть ли ограничения у вывода.

## Рекомендуемый narrative summary

После существенного эксперимента записывать в `RESEARCH_JOURNAL.md`:

- цель;
- метод;
- ключевые параметры;
- местоположение артефактов;
- краткий вывод;
- следующий шаг.

## Связь с гипотезами

Если эксперимент относится к гипотезе, нужно зафиксировать:

- ID гипотезы, если он есть;
- подтверждает ли эксперимент гипотезу, частично поддерживает или не поддерживает;
- какие ещё проверки остаются.

## Эталонная практика для engineering-side

Для инженерных прогонов ориентироваться на подход из `ML in Math/`:

- solve/evaluate/log;
- хранить артефакты запусков;
- сопоставлять логи с hypothesis/status documents;
- не ограничиваться одним числом score без контекста.

## Эталонная практика для math-side

Для математических вычислительных проверок ориентироваться на подход из `Math_Hypothese_AutoCheck_Witch_Agents/`:

- связывать эксперимент с гипотезой, инсайтом или proof attempt;
- фиксировать, является ли вычислительная проверка лишь evidence, а не доказательством;
- сохранять путь к дальнейшей формализации.
