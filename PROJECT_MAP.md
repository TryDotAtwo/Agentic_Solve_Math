# Project Map

## Роль корня

Корень `D:\Agentic_Solve_Math` теперь является координирующим слоем над двумя отдельными проектами. Он отвечает не за вычисления как таковые, а за:

- общую агентную память;
- стандарты документации;
- журнал промптов и действий;
- intake новых задач через `kaggle_intake/`;
- маршрутизацию работы между двумя проектами.

## Проект 1: `Math_Hypothese_AutoCheck_Witch_Agents/`

### Назначение

Исследовательская математическая среда для анализа математических гипотез при помощи LLM-агента, с упором на:

- Zettelkasten-подобную базу знаний;
- карточки гипотез, концепций, доказательств и инсайтов;
- мультимодельный анализ;
- вычислительную верификацию;
- Lean 4 формализацию.

### Сильные стороны

- Хорошо продумана долгосрочная память в `docs/`.
- Есть явный workflow от сырья к карточкам.
- Есть мультимодельная схема анализа.
- Есть задел для Lean и формальной верификации.

### Ограничения

- Мало единой автоматизации.
- Повторяемые вычисления во многом живут в ad hoc scripts.
- Небольшое покрытие тестами.
- Основной центр тяжести сейчас вокруг одной развитой гипотезы, а не общего engine.

### Когда направлять работу сюда

- Нужно разобрать математическую идею, статью или гипотезу.
- Нужно создать структурированные карточки знаний.
- Нужно провести мультимодельный анализ идеи.
- Нужно перевести вычислительные наблюдения в гипотезы и доказательные маршруты.
- Нужно готовить Lean-формализацию.

## Проект 2: `CayleyPy_Pancake/`

### Назначение

Инженерный Kaggle/algorithmic research проект, в текущем состоянии сфокусированный на **pancake problem**:

- solvers;
- beam search;
- notebook-derived algorithms;
- RL;
- experiment runners;
- submission evaluation;
- логирование и сравнение результатов;
- статья и гипотезы по улучшению качества.

### Сильные стороны

- Чёткие entry points: `main.py`, `run_best.py`, `run_experiment.py`.
- Хорошо оформленные docs для агентов.
- Есть журнал изменений и дисциплина синхронизации результатов.
- Есть накопленные артефакты экспериментов и логов.
- Есть тестовая структура и модульная организация `src/`.

### Ограничения

- Сейчас проект предметно узок и завязан на pancake problem.
- Внутренний `baseline/` уже содержит проектные оригиналы, поэтому корневой intake нельзя путать с ним.
- Часть теоретических идей ещё на уровне гипотез.
- Некоторые внешние артефакты и логи смешаны с рабочим деревом.

### Когда направлять работу сюда

- Нужно исследовать Kaggle solver pipeline.
- Нужно запускать, сравнивать и логировать методы.
- Нужно проверять гипотезы о качестве эвристик или RL.
- Нужно формировать submission, сравнивать score и документировать результаты.

## Правило маршрутизации

### Если вход — Kaggle competition

1. Прочитать файл из корневого `baseline/`.
2. Извлечь формальные требования соревнования и возможные baseline-источники.
3. Вести корневые журналы и память.
4. Для engineering-задач переходить в `ML in Math/`.
5. Для математических идей, инвариантов, гипотез и доказательных направлений переходить в `Math_Hypothese_AutoCheck_Witch_Agents/`.

### Если вход — математическая идея, статья, conjecture

1. Зафиксировать промпт и контекст в корневых журналах.
2. Основную обработку вести в `Math_Hypothese_AutoCheck_Witch_Agents/`.
3. Если идея приводит к вычислимой эвристике или solver improvement, создать связку с `ML in Math/`.

## Ключевые риски

- Нельзя смешивать корневой `baseline/` и `ML in Math/baseline/`: первый — intake новых задач, второй — архив/референс конкретного проекта.
- Нельзя переносить инженерные выводы в исследовательскую память без указания источников, логов и статуса проверки.
- Нельзя терять пользовательские промпты: они являются частью научной трассируемости и должны сохраняться отдельно.
- Нельзя обновлять локальные проектные docs без синхронизации с корневым журналом, если изменения значимы для общей программы исследований.
